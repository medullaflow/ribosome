# Design decisions

The rationale log for ribosome — *why* each significant design and tooling
choice was made, and the trade-offs weighed. It's contributor-facing context,
kept separate from [ARCHITECTURE.md](ARCHITECTURE.md) (the design reference) so
that page stays a map rather than a changelog.

Numbers are stable and never reused: a superseded or retired decision leaves a
gap rather than renumbering the rest, so a reference to "D20" means the same
thing forever.

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | JSON Schema is the source of truth; TS types are generated | The standard must be implementable in any language; a TS-first source would demote every other language to a translation. |
| D2 | `ribosome.json` is a standalone file, not a fragment of a host's schema | A de-facto standard can't be defined as "a hand-kept subset of consumer X". |
| D3 | The resolver is an `EnvironmentProvider` (env delta + optional activation hook) | Decouples from any one runtime manager; nix/asdf/devbox fit without leaking a per-tool `binPath` model. |
| D4 | Pure orchestrator; effects extracted to a thin persistence step | Testability — the core never touches the filesystem and returns the lockfile as data. |
| D5 | Shared runtime pool + isolated per-consumer views | Deduplicates installs without merging conflicting versions; native to how runtime managers already work. |
| D6 | MCP server runtimes are derived from the registry (`server.json`), not restated by the user | Exploits the MCP standard instead of asking the user to duplicate it. |
| D7 | Unpinned MCP runtimes fall back to the project's declared version, then a provider default | Ergonomic and deduplicated; the resolved version is recorded in the lockfile for reproducibility. |
| D8 | Named `registries` with a per-source `type` | Multiple registry protocols exist; `type` selects the adapter, decoupled from any specific URL. |
| D9 | Custom servers are declared as a full inline `server.json` | Reuses the whole resolution pipeline and gives a clean custom → published migration path. |
| D10 | Vendor and pin external schemas; never a live `$ref` | A standard whose meaning can change silently under a remote URL is not a standard. |
| D11 | `mcpServers` is a compatible superset of existing MCP config; `extends` reserved for import | Avoids being the N+1 competing format — conform on config, compete on runtime. |
| D13 | Split into two repos (the standard + this implementation) now, not at a later trigger | Doing the split while there's one author and a still-moving schema is cheaper than migrating more history later, and it settles the standard's independence — its own release cadence and license — at the repo level, matching how MCP, OpenAPI, and OCI separate spec from SDK. Trade-off accepted: cross-repo dev friction in exchange for reaching the target shape once. |
| D14 | Stay TypeScript; solve "no runtime required to run" by compiling a single binary (`bun build --compile`), not a Rust rewrite | A tool that provisions runtimes can't demand one to bootstrap, but the distribution problem and the language problem are separable: a compiled binary removes the runtime dependency at near-zero cost while keeping the dominant `npm install`-and-embed path intact. Revisit Rust only at a real trigger (embedding in a non-JS host), not speculatively. |
| D15 | One Linux runner cross-compiles every OS/arch; per-OS packaging format chosen by which tools are actually cross-platform | `bun build --compile --target` produces genuine native executables for all five targets from one runner. Windows (NSIS) and Linux (`.deb`/`.rpm`) packaging stay buildable there; a signed macOS `.pkg` needs Apple-only tooling and a paid account, so macOS ships as a `.zip` for now. Checksums and build-provenance attestation ship for every artifact regardless. |
| D17 | Two-tier release CI: cross-compile once, then a per-OS matrix that executes each already-built artifact before it's attached to the release | Compiling without error isn't evidence a cross-compiled binary runs; a separate matrix installs the real packaged artifact on each OS and runs a smoke `resolve`, gating the upload. Kept separate from the build tier so the higher runner cost is paid only on a published release. |
| D18 | Relicense from AGPL-3.0 to MPL-2.0 | ribosome is meant to be embedded in other products, including closed-source ones; strong copyleft works against that. MPL's file-level copyleft still returns changes to ribosome's own files while leaving the combined work unaffected — the protection actually wanted. Cheapest to do while the sole author holds 100% copyright and nothing is published. |
| D19 | `OfficialMcpRegistry` hand-types the registry's HTTP response envelope instead of generating a client from its OpenAPI spec | The one endpoint this read-only adapter needs has a 3-field envelope whose payload is already the vendored `McpServerJson` type; a codegen pipeline for that would be disproportionate. Revisit if the adapter ever needs a wider slice of the API. |
| D20 | Registry/inline/process sources normalize to a discriminated `ResolvedMcpServerRef`; runtime-derivation logic lives in `orchestrator/`, not `adapters/` | Process servers structurally have no `server.json`, so a single merged shape would invent fields that don't exist; a discriminated union is honest about that. The pure mapping logic sits in the orchestrator because it's registry-agnostic, and placing it under `adapters/` would force a forbidden import edge. |
| D21 | `deriveLaunch()` supports only `npm`/`pypi` packages plus remotes for now; unsupported packages and value-less arguments fail loudly | Scope set from a live 100-server registry sample, not assumption. `oci` is excluded outright because a container's env-var handling is a real unanswered design question, not a patch. Throwing on a missing launch token upholds the fail-at-validation-time pitch. |
| D22 | Multi-registry support needs no per-vendor adapter; the real gap was per-source auth headers | The registry ecosystem is federated — there's no enumerable list to hardcode, and adapter dispatch already keys on `type`, not URL. The one real gap was auth headers, added as `RegistrySource.auth` (values read from env vars, never stored) and failing via a typed `MissingRegistryCredentialError` before any request. |
| D23 | `FileMcpRegistry` reuses the official registry's `{ servers: [...] }` bulk-list envelope as its on-disk format | Makes `curl .../v0.1/servers > servers.json` directly usable offline and reuses the same validation path — one fewer format to invent or keep in sync. "Latest" resolution uses a minimal dot-numeric comparator, proportionate to a local use case. |
| D24 | Prove multi-source dispatch + auth with a local throwaway HTTP server, not a real third-party subregistry | What matters architecturally is dispatch across two same-type sources with the auth path exercised — not that a specific external registry is reachable. A local `node:http` server proves that with no external signup dependency. |
| D25 | `process` entries reuse the project's own pool view plus their own `env`; `activationHook` is stripped from lockfile `Environment` values | A `process` entry has no packages to derive from but "runs inside the project pool's environment," so it gets the project's composed view, not an empty one. `materialize()` runs resolution and provisioning before deciding to throw, so failures from both phases surface together. |
| D26 | The `inline` branch validates `entry.server` with non-throwing `checkMcpServerJson()`, folding failures into the same aggregated path | Nothing runs the manifest through validation earlier, so a malformed inline `server.json` would otherwise fail confusingly deep in the pipeline. Reuses the same check the registry adapters already run — one validation path, not a second. |
| D27 | `writeLockfile()` lives in `orchestrator/lockfile-writer.ts`, not behind a port/adapter | The lockfile has exactly one format and one target; a port for a single fixed implementation would be an abstraction with no second consumer. `materialize()` already returns the lockfile as a value, so testability is unaffected. |
| D28 | The convergence check also exercises the real `MiseEnvironmentProvider`, not just the real registry adapter | `materialize()` can't run without *some* environment provider, so "no test doubles in the path" necessarily includes it. Validates the written lockfile with the standard's own `validateLockfile()`, closing all pipeline steps in one run. |
| D29 | Pool location (`pool.dir` → `MISE_DATA_DIR`) and lifecycle (tracking + `prune()`) are built entirely on the backend's own mechanisms | mise already reference-counts tracked configs across projects, so a bespoke registry would be strictly worse. Fixing a real gap (a bare install was prunable by an unrelated `mise prune`) required routing through mise's default config discovery; `prune()` stays an explicit caller action, never a side effect of resolve. |
| D30 | The tests-inclusive typecheck is a second `noEmit` tsconfig, not a widened `include` on the base one | The base config's `rootDir`/`outDir` are load-bearing for the real build; widening its include would emit test files into `dist/`. A second, `extends`-ing config checks `src/` + `bin/` + `test/` (the shipped CLI entry had zero compiler coverage before) without disturbing the build. |
| D31 | Secret scanning runs the gitleaks CLI directly via a pinned binary, not the `gitleaks-action` wrapper | The action requires a paid license for org-owned repos; the MIT-licensed CLI has no such gate and gives the identical scan. |
| D32 | Dependency scanning is two complementary mechanisms — `dependency-review-action` (per-PR) plus `dependabot.yml` (scheduled) | The PR check covers the merge path; Dependabot's update PRs make routine bumps a reviewable event. Both are free for this public repo via GitHub's native dependency graph. |
| D33 | `check-dco.js` exempts Dependabot's own commits by author email; no workflow rewrites commits to add sign-off | Dependabot authors and signs off under different addresses, failing DCO out of the box; its original commit is already GitHub-verified, so nothing needs rewriting once the check accepts it. DCO certifies a human's right to contribute — accountability for a bot bump lands on the reviewer who merges it. |
| D34 | The TypeScript 7.0 migration landed via `@typescript/typescript6` for the architecture-rules parser, not a rewrite against `typescript/unstable/ast` | TS 7.0 ships no programmatic API at all; the `unstable` subpath exists but its maintainer says not to build tooling on it yet. `@typescript/typescript6` is the sanctioned bridge (the full classic API), used only by that one script while `tsc` itself runs on 7.0. |
| D35 | Added `bun audit` and CodeQL; declined SBOM and dependency-license scanning as near-noise at one dependency | `bun audit` closes a real gap (CVEs disclosed after a dependency is already in the lockfile); CodeQL catches a different bug class than the linter on real shell-out/path-handling code. SBOM and license scanning have almost nothing to scan with a single dependency; revisit once real third-party deps exist. |
| D36 | `CODEOWNERS` gates every PR on maintainer review even in a single-maintainer repo | Most code here is agent-authored under direction, so the human-judgment gate needed to be explicit. Until a second contributor exists, every merge goes through a logged admin bypass — a visible, audited accountability event — and the gate becomes literal the moment a second owner is added. |
| D37 | The coverage floor is per-file at 80% lines, `test/**` excluded | A per-file floor stops a new untested file hiding behind a high repo-wide average; 80% sits just below the lowest real file so ordinary edits don't trip it. (The enforcement mechanism was later re-implemented on `node --test` — see D50.) |
| D39 | Mutation testing runs via Stryker's generic `command` runner over a fixed deterministic subset, in-place, advisory not gating | No official Stryker bun runner exists, and the copy-to-sandbox mode crashes on TS 7; `inPlace` sidesteps both. The subset is the fixture-based pipeline logic — the network/mise files can't be meaningfully mutation-tested. Advisory because a hard threshold on today's score would be either blocking or meaningless. |
| D40 | npm publishing uses OIDC trusted publishing, gated on a published GitHub Release | No `NPM_TOKEN` ever exists — the runner exchanges a short-lived OIDC token for publish rights. `npm publish` stays on the npm CLI (bun doesn't support OIDC publishing yet); the first-ever version needs a one-time manual publish, documented in the workflow. |
| D41 | The publish workflow's `smoke-test` installs the published tarball and exercises its real exports | Existing tests run against `dist/` directly and never go through what `npm install` resolves, so a bad `files`/`main`/`exports` could ship undetected. It checks exactly the documented public API and retries the install to absorb registry propagation lag. |
| D42 | Shared CLI logic lives in `src/cli.ts` (tsc → `dist/cli.js`, the npm `bin`); the compiled-binary entry delegates to it | The old binary-only entry couldn't serve as an npm `bin` because tsc's build is scoped to `src/`. One shared implementation avoids two drifting copies; the move needed only import-path and shebang changes, since the code was already plain Node API. |
| D43 | A vendored, pinned mise binary ships in every packaged release, in a package-private dir; `resolveMiseBinary()` prefers it over `PATH`, with an env override | A "zero-setup" installer can't fail on a missing mise the first time it shells out. Placement is package-private (never a shared `/usr/bin`) to avoid clobbering a user's own mise. Resolution order: `RIBOSOME_MISE_BIN` > a bundled sibling of the running binary > bare `mise` — so library consumers (no bundled binary) are unaffected and a patched system mise is never unreachable. Checksums are pinned; the large binaries are fetched at release-build time, never committed; weekly drift detection keeps the pin fresh. |
| D44 | Per-OS packaging is three `node` scripts run from a matrix, all on one Linux runner | Every format places the ribosome binary and its `mise-bundled/` sibling together (D43's one constraint). `.rpm` for arm64 can't cross-build on x86_64 (a static architecture gate, not a runtime probe), so that leg runs `rpmbuild` inside an emulated aarch64 container where it's the native case. Windows PATH integration uses built-in PowerShell via `nsExec`, avoiding any NSIS plugin. Install-and-run is left to the separate verification tier. |
| D45 | Checksums and attestation were built together with the release-triggered mechanism in one `release.yml` that calls the packaging workflow | Building them against the old dispatch-only trigger and re-wiring later would redo, not reuse, the work. A `SHA256SUMS` manifest plus a provenance attestation over a glob cover every artifact in one `finalize-release` job — kept a separate job precisely so the future verification tier slots in as a dependency before upload. |
| D46 | `release.yml` has a `workflow_dispatch` dry-run: an empty input runs package + checksum + attest for real but skips upload; a tag pointing at a draft release exercises upload too | Release-gated workflows can otherwise only be rehearsed by cutting a real release — which would also trigger a real npm publish. A draft release doesn't fire `release: published`, so uploading to one via manual dispatch exercises the real upload path with zero publish side effect. Attestation runs on every dry run, since it's the part most worth rehearsing. |
| D47 | `OfficialMcpRegistry.resolve()` retries connection failures/timeouts and 5xx up to 3 attempts with incremental backoff; 4xx and a resolved 404 are never retried | The live registry was directly observed timing out and then answering within a second — genuinely transient. This is production resilience, not just a CI fix, since a real `resolve` hits the same registry. A 404 is the registry answering definitively, so retrying it would be slower, not more correct. |
| D48 | `launch-mapping.test.ts`'s `fetchServer()` carries its own copy of the retry-with-backoff around its raw `curl` call | That file deliberately bypasses the adapter (it tests mapping logic, not HTTP behavior), so it never inherited D47's retry. No shared helper: curl's exit code can't distinguish a retryable 5xx from a definitive 404, so this copy retries unconditionally — safe only because every server it queries is a known-good fixture. |
| D49 | The docs site is an Astro + Starlight project under `docs/site/` with its own dependency tree, deployed by a standalone workflow | Astro + Starlight was chosen over VitePress for first-class sitemap generation (the indexability requirement), plus built-in full-text search and sidebar nav. Its own `package.json` keeps that dependency tree out of `src`/`test` work. Content is hand-authored — the CLI page embeds real captured `--help` output so it can't drift. |
| D50 | Test runner switched from `bun test` to real `node --test`: the runner compiles `test/` to CommonJS via `tsc`, symlinks in referenced repo paths, and runs under `--experimental-test-coverage`; the CI retry-on-known-bug wrapper is removed | The flake was a bug in Bun's *reimplementation* of `node:test`; every test file always imported from `node:test` itself, so running on the runtime that implements it natively is a fix, not a workaround. Compiling via tsc (not Node's type-stripper) desugars parameter properties and resolves extensionless imports; symlinks preserve `__dirname`-relative paths with zero source changes. The coverage floor is lines-only because V8 coverage has no statements axis. |
| D51 | The registry per-attempt timeout was raised 10s→20s and backoff 500ms→1000ms, with every dependent test budget recalculated | A sustained degraded period (a real `HTTP 200` taking ~12-13s, not a brief stall) fired the old 10s timeout before a healthy-but-slow response could arrive, defeating the retries. 20s leaves margin above the observed latency. A first pass sized test timeouts for 2 attempts; corrected to the adapter's full 3-attempt worst case after a live test timed out at the tighter bound. Attempt count was deliberately not raised — the problem was time per attempt, not too few attempts. |
| D52 | `FallbackMcpRegistry` wraps any `McpRegistry` with an ordered list of mirror URLs, tried only on `RegistryUnreachableError`; the CLI wires it in when `RIBOSOME_REGISTRY_MIRRORS` is set | Any definitive error (not-found, invalid, missing credential) never triggers a mirror. The named "Anthropic/Microsoft registries" turned out not to be distinct hardcodable endpoints, so this ships as pure mechanism with zero default mirrors. It composes at the wiring layer, not the manifest schema — redundant mirrors of one catalog are an operational decision, distinct from `registries.sources`' multiple *distinct* catalogs, so auto-cascading through those was rejected as a correctness risk. |
